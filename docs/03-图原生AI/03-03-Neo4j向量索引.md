## 一、向量嵌入基础概念

### 1.1 什么是向量嵌入

向量嵌入（Vector Embedding）是指将现实世界中的离散对象——文本、图像、节点、关系——映射为一个固定维度的实数向量，使得"语义或结构上相近"的对象在向量空间中距离也相近。这一映射通常由嵌入模型（Embedding Model）学习得到，是连接符号世界与连续数学世界的桥梁。

以文本为例，句子"图数据库擅长处理关联数据"和"Neo4j 善于管理关系型信息"在字面上差异较大，但语义相近，因此它们的向量嵌入在高维空间中彼此接近；而"今天天气不错"的向量则与二者相距较远。这种"语义相近则向量相近"的性质，是向量检索的基础。

### 1.2 嵌入的关键属性

理解向量嵌入，需要把握几个关键属性：

| 属性 | 说明 | 典型取值 |
|------|------|---------|
| 维度（Dimension） | 向量的实数个数 | 384、768、1536、3072 |
| 相似度度量（Similarity） | 衡量两向量相近程度的方式 | 余弦相似度、欧氏距离、点积 |
| 嵌入模型 | 生成向量的模型 | OpenAI text-embedding-3、HuggingFace all-MiniLM、Cohere embed |
| 归一化（Normalization） | 向量是否缩放为单位长度 | 是/否，影响相似度计算 |

在 GraphRAG 与图原生 AI 场景中，最常见的相似度度量是余弦相似度（Cosine Similarity），它衡量两向量夹角的余弦值，对向量的绝对长度不敏感，适合比较语义方向。

### 1.3 图场景下的嵌入对象

在 Neo4j 图数据库中，可以产生向量嵌入的对象包括：

- **节点嵌入**（Node Embedding）：将节点的属性文本（如标题、摘要）编码为向量，或用图嵌入算法（Node2Vec、FastRP）将节点的拓扑结构编码为向量；
- **文本块嵌入**（Chunk Embedding）：将节点关联的长文本切分为块，每块独立编码为向量，常用于文档问答；
- **关系嵌入**（Relation Embedding）：将关系类型或关系属性编码为向量，用于关系预测等任务。

本章聚焦于"节点属性文本的向量嵌入 + 向量索引"，这是 GraphRAG 向量检索部分最常用的模式。

## 二、Neo4j 向量索引功能概述

Neo4j 从 5.11 版本开始引入原生向量索引（Vector Index），使向量检索能力内建于图数据库之中。这一功能的引入，是 Neo4j 迈向图原生 AI 的关键一步——它使得向量嵌入与图结构可以共存于同一数据库，无需外接向量数据库。

### 2.1 核心过程：db.index.vector.queryNodes

Neo4j 向量索引的核心查询过程是 `db.index.vector.queryNodes`。该过程接收一个查询向量，在指定的向量索引上执行近似最近邻（Approximate Nearest Neighbor，ANN）搜索，返回最相似的若干节点及其相似度分数。

```cypher
CALL db.index.vector.queryNodes(
    'index_name',   // 向量索引名称
    10,             // 返回的邻居数量 (topK)
    $queryVector    // 查询向量
)
YIELD node, score
RETURN node.title AS title, score
ORDER BY score DESC;
```

返回结果中，`node` 是匹配到的图节点，`score` 是相似度分数（余弦相似度下取值范围为 -1 到 1，越接近 1 越相似）。由于返回的是图节点，可以立即在该节点上进行图遍历，这正是 Neo4j GraphRAG 混合检索的基础。

### 2.2 向量索引的底层算法

Neo4j 向量索引默认采用基于近邻图的近似最近邻算法，在检索精度与速度之间取得平衡。相较于精确 KNN（需要与所有向量逐一比较，复杂度 O(n)），近似最近邻在牺牲极小精度的前提下，将检索复杂度降至近 O(log n)，适合大规模向量集合。

| 算法类型 | 精度 | 速度 | 适用规模 | Neo4j 是否采用 |
|---------|------|------|---------|--------------|
| 精确 KNN（暴力） | 100% | 慢 | 小规模（<10 万） | 否（默认） |
| 近邻图 ANN（如 HNSW 思想） | 高（可调） | 快 | 大规模（百万级） | 是（默认） |

## 三、向量索引的创建与配置

### 3.1 创建向量索引

在 Neo4j 中创建向量索引使用 `CREATE VECTOR INDEX` 语句。以下示例为 `Paper` 节点的 `embedding` 属性创建一个基于余弦相似度的向量索引：

```cypher
CREATE VECTOR INDEX paper_embedding_index
FOR (p:Paper) ON (p.embedding)
OPTIONS {
    indexConfig: {
        `vector.dimensions`: 1536,
        `vector.similarity_function`: 'cosine'
    }
};
```

关键配置项说明：

| 配置项 | 含义 | 取值要求 |
|-------|------|---------|
| `vector.dimensions` | 向量维度 | 必须与嵌入模型输出维度一致 |
| `vector.similarity_function` | 相似度函数 | `cosine`（默认）、`euclidean` |

需要注意，`vector.dimensions` 必须严格等于所使用嵌入模型的输出维度。例如 OpenAI `text-embedding-3-small` 输出 1536 维，则索引维度必须设为 1536；HuggingFace `all-MiniLM-L6-v2` 输出 384 维，则索引维度设为 384。维度不匹配会导致索引创建失败或检索异常。

### 3.2 查看与删除向量索引

```cypher
// 查看所有索引（含向量索引）
SHOW INDEXES YIELD name, type, entityType, labelsOrTypes, properties
WHERE type = 'VECTOR'
RETURN name, labelsOrTypes, properties;

// 删除向量索引
DROP INDEX paper_embedding_index;
```

### 3.3 索引状态与同步

向量索引创建后会进入"同步"（populating）状态，待所有现有节点的向量被索引后转为"在线"（online）状态。新写入的向量会自动增量同步到索引。可通过以下查询检查索引状态：

```cypher
SHOW INDEXES YIELD name, state, populationPercent
WHERE name = 'paper_embedding_index'
RETURN name, state, populationPercent;
```

## 四、嵌入模型集成

向量索引本身只负责"存储与检索向量"，向量的生成需要由嵌入模型完成。Neo4j 不内置嵌入模型，而是通过与外部嵌入服务集成来生成向量。常见的集成方案包括 OpenAI、HuggingFace、Cohere 等。

### 4.1 嵌入模型对比

| 嵌入模型 | 提供方 | 维度 | 特点 | 适用场景 |
|---------|-------|------|------|---------|
| text-embedding-3-small | OpenAI | 1536 | 性价比高、英文为主 | 通用英文文本 |
| text-embedding-3-large | OpenAI | 3072 | 高精度、可降维 | 高精度需求 |
| all-MiniLM-L6-v2 | HuggingFace | 384 | 开源免费、可本地部署 | 中文/多语言、低成本 |
| bge-large-zh | 智源/BAAI | 1024 | 中文效果好、开源 | 中文专业领域 |
| embed-english-v3 | Cohere | 1024 | 英文优化、多模态扩展 | 英文企业应用 |

对于中文图书情报场景，推荐使用 `bge-large-zh` 等中文优化的开源模型，可在本地部署以降低成本并保护数据隐私。

### 4.2 用 Neo4j GenAI 插件生成嵌入

Neo4j 提供了 APOC 与 GenAI 插件，可在 Cypher 中直接调用外部嵌入 API，实现"查询时生成嵌入"的便捷流程。

```cypher
// 通过 APOC 调用 OpenAI 嵌入 API（需配置 API Key）
MATCH (p:Paper) WHERE p.embedding IS NULL
CALL apoc.ml.openai.embedding([p.title + '. ' + p.abstract], 'your-api-key')
YIELD index, text, embedding
SET p.embedding = embedding;
```

### 4.3 用 Python 预生成嵌入后写入

另一种常见方案是在 Python 中预先生成嵌入，再批量写入 Neo4j。这种方式更灵活，适合离线批处理。

```python
from neo4j import GraphDatabase
from sentence_transformers import SentenceTransformer

# 加载 HuggingFace 嵌入模型
model = SentenceTransformer('all-MiniLM-L6-v2')
driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))

def write_embeddings():
    with driver.session() as session:
        # 读取待嵌入的节点
        papers = session.run("MATCH (p:Paper) WHERE p.embedding IS NULL RETURN id(p) AS id, p.title AS title, p.abstract AS abstract").data()
        
        for p in papers:
            text = f"{p['title']}. {p['abstract']}"
            emb = model.encode(text).tolist()  # 384 维向量
            session.run("MATCH (p) WHERE id(p) = $id SET p.embedding = $emb", id=p['id'], emb=emb)

write_embeddings()
```

## 五、节点上存储 embedding 的实现方式

在 Neo4j 中，向量嵌入作为节点的普通属性存储。这一设计看似简单，却具有重要的架构意义。

### 5.1 属性存储模式

```cypher
// 创建带嵌入属性的节点
CREATE (p:Paper {
    title: 'GraphRAG: From Theory to Practice',
    abstract: 'This paper explores graph retrieval augmented generation...',
    embedding: [0.012, -0.034, 0.078, ... /* 1536 维 */]
});
```

向量就是一个浮点数列表（`LIST<FLOAT>`），与标题、摘要等普通属性并列存储。这种"嵌入即属性"的设计带来几个好处：

1. **事务一致**：嵌入与节点其他属性在同一事务中写入，不会出现"节点存在但嵌入缺失"的不一致状态；
2. **图遍历无缝衔接**：向量检索返回的节点可直接参与 Cypher 图遍历，无需跨库关联；
3. **统一权限与备份**：嵌入随图数据库一起备份、迁移、权限控制，运维链路统一。

### 5.2 嵌入属性的更新策略

当节点文本属性更新时，对应的嵌入也需更新以保持一致性。推荐使用触发器或在应用层封装更新逻辑：

```cypher
// 更新文本时同步更新嵌入
MATCH (p:Paper {title: 'GraphRAG: From Theory to Practice'})
SET p.abstract = 'Updated abstract content...'
WITH p
CALL apoc.ml.openai.embedding([p.title + '. ' + p.abstract], 'your-api-key')
YIELD embedding
SET p.embedding = embedding;
```

### 5.3 大文本的分块嵌入

当节点关联的文本很长（如全文论文）时，单次嵌入可能超出模型 token 上限或丢失细节。此时可采用分块（Chunking）策略：将长文本切分为多个块，每个块作为独立节点存储嵌入，并通过 `DESCRIBES` 关系关联到主节点。

```cypher
// 主节点
CREATE (p:Paper {id: 'p001', title: '...'})

// 文本块节点（每个块独立嵌入）
CREATE (c1:Chunk {text: '第一段...', embedding: [...]})
CREATE (c2:Chunk {text: '第二段...', embedding: [...]})

// 关联
CREATE (c1)-[:DESCRIBES]->(p)
CREATE (c2)-[:DESCRIBES]->(p);

// 为文本块创建向量索引
CREATE VECTOR INDEX chunk_embedding_index
FOR (c:Chunk) ON (c.embedding)
OPTIONS { indexConfig: { `vector.dimensions`: 1536, `vector.similarity_function`: 'cosine' } };
```

这种分块模式在文档问答（Document QA）场景中非常常见，检索时先命中文本块，再通过关系回溯到主节点及其关联实体。

## 六、向量相似度搜索示例

### 6.1 基本 KNN 查询

以下示例展示一个完整的"查询文本→生成嵌入→向量检索→图遍历"流程：

```cypher
// 假设已通过应用层将查询文本转为向量 $queryVector
// 第一步：向量检索召回 top-5 相似论文
CALL db.index.vector.queryNodes('paper_embedding_index', 5, $queryVector)
YIELD node, score
WITH node AS paper, score
// 第二步：从相似论文出发，遍历其作者与主题
MATCH (paper)-[:WRITT_BY]->(a:Author)
OPTIONAL MATCH (paper)-[:ABOUT]->(t:Topic)
RETURN paper.title AS title, score, 
       collect(DISTINCT a.name) AS authors,
       collect(DISTINCT t.name) AS topics
ORDER BY score DESC;
```

这条查询同时完成了向量检索与图遍历，是 GraphRAG 混合检索的最小可运行示例。

### 6.2 带过滤的向量检索

`db.index.vector.queryNodes` 本身不支持复杂过滤条件，但可以在 YIELD 之后用 WHERE 子句对结果过滤。若需在检索前过滤（pre-filtering），可借助 Neo4j 的多类型索引能力。

```cypher
// 向量检索后按年份过滤
CALL db.index.vector.queryNodes('paper_embedding_index', 20, $queryVector)
YIELD node, score
WITH node AS paper, score
WHERE paper.year >= 2020
RETURN paper.title, paper.year, score
ORDER BY score DESC
LIMIT 5;
```

### 6.3 相似度分数解读

余弦相似度分数的取值与含义：

| 分数范围 | 含义 | 典型场景 |
|---------|------|---------|
| 0.9 - 1.0 | 高度相似 | 同一主题、近义表达 |
| 0.7 - 0.9 | 较为相似 | 相关主题 |
| 0.5 - 0.7 | 弱相关 | 边缘相关 |
| < 0.5 | 不相关 | 不同领域 |

在实际应用中，通常会设置一个相似度阈值（如 0.7）过滤掉弱相关结果，以提升检索精度。

## 七、向量索引与传统全文索引对比

Neo4j 同时支持全文索引（Full-text Index）与向量索引，二者在检索原理与适用场景上差异显著。理解它们的区别，有助于在 GraphRAG 中合理组合两种检索能力。

| 对比维度 | 全文索引（Full-text Index） | 向量索引（Vector Index） |
|---------|---------------------------|------------------------|
| 检索原理 | 词项匹配（基于倒排索引、BM25） | 语义相似度（基于向量空间） |
| 匹配粒度 | 精确词/词组 | 语义整体 |
| 是否理解同义词 | 否（除非配置同义词表） | 是（嵌入模型隐式捕获） |
| 跨语言能力 | 弱 | 强（多语言嵌入模型） |
| 模糊匹配 | 有限（通配符、模糊查询） | 强（语义相近即匹配） |
| 可解释性 | 高（命中词项可见） | 低（相似度为黑盒） |
| 索引构建成本 | 低（分词即可） | 中（需调用嵌入模型） |
| 适合场景 | 精确实体名、关键词检索 | 语义相似、自然语言问答 |

### 7.1 两种索引的协同

在 GraphRAG 中，全文索引与向量索引并非二选一，而是协同互补：

- **全文索引**适合精确匹配实体名、DOI、ISBN 等标识符，以及术语精确检索；
- **向量索引**适合自然语言问句的语义匹配，处理用户多样化的表达方式。

一个典型的协同流程是：先用全文索引做精确召回（如匹配作者名），再用向量索引做语义扩展（如召回语义相近论文），最后融合结果。这正是混合检索的思想，将在下一章详细展开。

### 7.2 创建全文索引示例

```cypher
// 创建全文索引
CREATE FULLTEXT INDEX paper_fulltext
FOR (p:Paper) ON EACH [p.title, p.abstract];

// 全文检索
CALL db.index.fulltext.queryNodes('paper_fulltext', 'graph database knowledge')
YIELD node, score
RETURN node.title, score
ORDER BY score DESC LIMIT 5;
```

## 八、与图书情报领域的关联

向量索引技术对图书情报领域的信息检索实践具有直接的迁移价值。

### 8.1 从布尔检索到语义检索的延续

图书情报领域的信息检索经历了从布尔检索（Boolean Retrieval）到向量空间模型（Vector Space Model，如 TF-IDF）再到语义检索（Semantic Retrieval）的演进。Neo4j 向量索引可以被视为这一演进在图数据库内的工程实现：

| 检索范式 | 代表方法 | Neo4j 对应 |
|---------|---------|-----------|
| 布尔检索 | AND/OR/NOT 逻辑 | Cypher WHERE 子句 |
| 向量空间模型 | TF-IDF + 余弦相似度 | 全文索引（BM25） |
| 语义检索 | 神经嵌入 + 余弦相似度 | 向量索引 |

### 8.2 向量检索与知识图谱的融合价值

对图书情报领域而言，Neo4j 向量索引的独特价值在于"向量与图共存"。传统语义检索系统（如基于 Elasticsearch 的语义搜索）只能返回文档，无法揭示文档间的知识关联；而 Neo4j 在向量检索返回节点后，可立即通过图遍历揭示该文献的作者、主题、引文网络等关联结构，使检索结果从"孤立的相似文档"升级为"有上下文的知识网络"。

| 检索能力 | 传统语义检索系统 | Neo4j 向量索引 + 图 |
|---------|----------------|--------------------|
| 语义相似文档 | 支持 | 支持 |
| 文档作者/主题关联 | 不支持 | 支持（图遍历） |
| 引文链追溯 | 不支持 | 支持（多跳遍历） |
| 实体消歧 | 弱 | 强（图中实体唯一） |
| 检索结果可解释性 | 弱（仅相似度） | 强（关联路径可见） |

### 8.3 对文献检索服务的启示

在现代数字图书馆与文献数据库中，用户往往用自然语言描述信息需求（如"用图神经网络做文献推荐的方法"），而非精确的关键词。向量索引使这类自然语言查询能够被语义理解并匹配到相关文献，而图遍历则进一步揭示文献间的引用、共被引、主题关联，为用户提供"检索—发现—溯源"一体化的知识发现体验。这正是图书情报领域从"文献检索"走向"知识发现"的技术路径。
