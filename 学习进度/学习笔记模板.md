## 二、填写规范与建议

1. **编号连续**：笔记按时间顺序编号（#1、#2……），便于检索与回看。
2. **先复述后引用**：核心概念区务必用自己的话复述，照抄原文记忆留存率低。
3. **代码必留**：凡涉及 Cypher / 命令 / 配置，均粘贴到代码块，构建个人"可复用片段库"。
4. **疑问不过夜**：疑问区标注优先级，下次学习前先回顾并尝试解答，解答后改为 `[x]`。
5. **领域关联是核心**：每条笔记都应尝试回答"这对我研究图书情报问题有什么用"，这是本知识库的设计初衷。
6. **实践留痕**：实践记录区保留命令与结果摘要，便于复盘与复现。
7. **定期回顾**：每完成一个篇目，回看该篇全部笔记，提炼一篇"篇目小结"。

> **小技巧**：可在每条笔记顶部用标签标注主题，如 `#属性图` `#Cypher` `#GDS` `#图嵌入` `#GraphRAG` `#图像知识图谱`，便于后续按主题检索。

### 笔记 #1 ｜ 属性图模型：把"关系"当作一等公民

- **笔记日期**：2025 年 9 月 18 日
- **所属篇目**：第 1 篇 · 图数据库基础
- **阅读文档**：
  - [x] 01-02 属性图模型 (`docs/01-foundations/01-02-property-graph-model.md`)
  - [x] 01-01 图论基础概念（复习）
- **投入时长**：约 60 分钟
- **难度感受**：入门（体感偏进阶，概念多但可理解）

#### 1. 核心概念记录

- **属性图四要素**：节点（Node）、标签（Label）、关系（Relationship）、属性（Property）。节点可有多个标签，关系必须有方向和类型。
- **关系是一等公民**：与传统 RDBMS 用外键隐含表达关系不同，图数据库把关系显式存储为实体，可带属性、可被直接遍历。
- **无索引邻接（Index-Free Adjacency）**：节点的邻接关系物理上就近存储，遍历邻居不需要回表查索引，因此图遍历复杂度只与子图规模相关，与全图规模无关——这是图数据库对比关系库的核心优势。
- **建模思维转变**：从"把实体拆成表、用外键连接"转向"先把实体和关系画出来，再落地为节点与边"。

#### 2. 关键代码 / Cypher 片段

```cypher
// 节点 + 关系 + 属性的一次性创建
CREATE (img:Image {image_id: "IMG001", filename: "cat.jpg", width: 1920, height: 1080})
CREATE (obj:Object {object_id: "OBJ01", name: "cat", category: "animal"})
CREATE (img)-[:IMAGE_DEPICTS_OBJECT {confidence: 0.96}]->(obj);
```

#### 3. 疑问与待深入

- [ ] (中) 一个节点可以有多个标签，多标签在查询性能上是否有影响？何时该用多标签而非子标签？
- [ ] (低) 关系的方向在语义上有时是对称的（如"相似"），Neo4j 如何高效处理无向查询？

#### 4. 与图书情报（LIS）领域关联思考

- **可迁移场景**：叙词表/本体（Skos、本体论）天然是概念关系图，可用 `Concept` 节点 + `broader`/`narrower`/`related` 关系直接建模，比关系库的"词表表"直观得多。
- **类比映射**：Dublin Core 的 15 个核心元素可视为图像节点的属性；而图像内容层面的物体、场景则需下沉为独立节点——这正是图像知识图谱对元数据编目的扩展。
- **潜在研究问题**：能否用属性图统一表达"文献元数据 + 引用关系 + 主题概念网络"，构建一个一体化的文献知识图谱？

#### 5. 实践记录

- **执行环境**：Neo4j Desktop（本地）
- **操作与结果**：在 Neo4j Browser 执行了上面的 CREATE 语句，成功创建 1 个 Image 节点、1 个 Object 节点和 1 条 DEPICTS 关系，图形视图直观显示了二者连接。
- **踩坑与解决**：首次忘了加分号，整段语句未执行；补上 `;` 后正常。

### 笔记 #2 ｜ Cypher 模式匹配与 SQL JOIN 的思维差异

- **笔记日期**：2025 年 9 月 20 日
- **所属篇目**：第 1 篇 · 图数据库基础
- **阅读文档**：
  - [x] 01-04 Cypher 查询语言详解
  - [x] 01-05 Cypher 实战示例集（部分）
- **投入时长**：约 90 分钟
- **难度感受**：进阶

#### 1. 核心概念记录

- **声明式模式匹配**：Cypher 用 `()` 表示节点、`--` / `-->` 表示关系，把"要找什么样的子图"画出来，由引擎决定怎么找；SQL 的 JOIN 则是"怎么把表拼起来"的过程式思维。
- **核心子句链**：`MATCH`（找模式）→ `WHERE`（过滤）→ `WITH`（传递中间结果）→ `RETURN`（投影输出）。`WITH` 是流水线的关键，相当于 SQL 的子查询/CTE。
- **`MERGE` vs `CREATE`**：`MERGE` 是"存在则复用、不存在则创建"，幂等；但 `MERGE` 全量匹配时若部分属性已存在会复用旧节点，导致属性未更新——需配合 `ON CREATE SET` / `ON MATCH SET`。
- **`OPTIONAL MATCH`**：类似 SQL 的 LEFT JOIN，匹配不到时对应变量为 `null`，适合"统计所有图像，顺便看它们有没有标注"这类场景。

#### 2. 关键代码 / Cypher 片段

```cypher
// 查找包含"猫"和"沙发"两类物体的图像（模式匹配 + 聚合）
MATCH (img:Image)-[:IMAGE_DEPICTS_OBJECT]->(o1:Object {name: "cat"})
MATCH (img)-[:IMAGE_DEPICTS_OBJECT]->(o2:Object {name: "sofa"})
RETURN img.filename AS image, count(*) AS matchCount
ORDER BY matchCount DESC;
```

#### 3. 疑问与待深入

- [ ] (高) `WITH` 与 `RETURN` 都能做聚合，二者的边界到底在哪？何时必须用 `WITH`？
- [ ] (中) 多重 `MATCH` 与用逗号分隔的多个模式（`MATCH (a), (b)`）在语义和性能上有何区别？

#### 4. 与图书情报（LIS）领域关联思考

- **可迁移场景**：查找"同时引用了 A 和 B 两篇文献"的论文，本质就是上面的"包含特定物体组合"模式，把 `IMAGE_DEPICTS_OBJECT` 换成 `CITES` 即可。
- **类比映射**：`OPTIONAL MATCH` 对应"列出所有期刊，并左连接其影响因子（可能缺失）"的 LEFT JOIN 场景。
- **潜在研究问题**：用 Cypher 模式匹配实现"研究主题共现网络"的构建——找出在摘要中共同出现某组主题词的文献。

#### 5. 实践记录

- **执行环境**：Neo4j Desktop（本地）
- **操作与结果**：在笔记 #1 建立的少量数据上跑了模式匹配查询；数据太少时结果不明显，决定后续执行 `01-create-image-graph.cypher` 注入完整样本数据再练。
- **踩坑与解决**：`WHERE` 写在 `MATCH` 后与写在关系模式内 `-[r WHERE ...]->` 行为不同，后者是关系模式谓词，注意区分。
